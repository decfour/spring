# 🛒 Commerce Service
> **Spring Boot와 Spring Data JPA를 활용한 이커머스 핵심 기능 구현 프로젝트**

## 🚀 Key Objectives
* **계층형 아키텍처**: Controller-Service-Repository 구조를 통한 책임 분리 및 비즈니스 로직 독립성 확보.
* **데이터 영속화(Persistence)**: 인터페이스 기반 설계를 바탕으로 인메모리(HashMap) 저장소에서 **MySQL로 전환 및 데이터 영속화**.
* **트랜잭션 기반 데이터 정합성 보장**: `@Transactional`을 활용한 서비스 레이어 원자성 확보 및 데이터 제어.
* **예외 처리 및 방어적 설계**: 사용자 조작 및 동시성 상황에서 발생할 수 있는 재고 부족과 데이터 왜곡을 백엔드 코어에서 차단.
* **멀티스레드 환경의 동시성 제어**: 다중 요청이 몰리는 환경을 시뮬레이션하고, 데이터 왜곡과 재고 정합성 문제를 백엔드 영역에서 차단.

## 🛠 Tech Stack
* **Framework**: Spring Boot 3.x
* **Language**: Java 17
* **ORM & Database**: Spring Data JPA, MySQL 8.x
* **Build**: Gradle
* **Template Engine**: Thymeleaf

## 📌 Main Features & Technical Achievements
### 1. 세션 기반 인증 구조 개선 (`HttpSession`)
* **보안 취약점 개선**: 기존 쿠키 기반 인증의 위조 위험을 인지하고, 서버 메모리에 추측 불가능한 **무작위 UUID(세션 Key)를 매핑**하는 방식으로 아키텍처 고도화.
* **인가(Authorization) 분리**: `HandlerInterceptor`를 도입하여 비로그인 사용자의 접근을 라우팅 단계에서 차단, 컨트롤러의 공통 인가 책임 제거.

### 2. 주문/장바구니 도메인 정합성 및 최적화
* **트랜잭션 원자성(Atomicity)**: '주문 생성'과 '장바구니 비우기'를 단일 트랜잭션으로 묶어 예외 발생 시 전 과정 롤백 보장.
* **영속성 컨텍스트 활용**: 무분별한 Update 쿼리 작성을 지양하고, JPA의 **변경 감지(Dirty Checking)**를 활용해 객체지향적 수정 반영.

### 3. 멀티스레드 동시성(Concurrency) 검증
* **Race Condition 시뮬레이션**: `ExecutorService`(스레드 풀)와 `CountDownLatch`를 활용하여 **100명의 사용자가 동시에 주문하는 환경의 테스트 케이스 구축**.
* **재고 무결성 확보**: 동시 다발적 요청 시 발생하는 데이터 유실 및 재고 음수(-) 현상을 방어하기 위해 애플리케이션 및 DB 레벨의 락(Lock) 전략을 분석 및 적용.

### 4. 코드 품질 및 예외 핸들링
* **NPE 원천 차단**: Java `Optional`과 람다식을 적극 활용하여 `null` 직접 반환 패턴 제거.
* **2중 검증 인프라**: 화면단(`th:max`)과 서버단(영속성 최신 재고 비교)의 **2중 검증 아키텍처**를 구축하고, `@ExceptionHandler`로 비즈니스 예외 처리 일원화.

## 🚨 Troubleshooting
### 1. 다중 요청(Race Condition) 발생 시 재고 데이터 무결성 붕괴
* **문제 상황**:
    * 단일 스레드 테스트 환경에서는 상품 주문 및 재고 차감 로직이 정상 작동하였으나, 멀티스레드 스트레스 테스트 수행 시 데이터 정합성이 깨지는 현상 발생.
    * 재고가 100개인 상품에 대해 **100개의 주문 요청이 동시에 인입되었음에도 최종 남은 재고가 0이 아니라 80~90개 등으로 남거나 데이터 유실**이 발생.

* **원인 분석**:
    * 여러 스레드가 동시 다발적으로 하나의 공유 자원(상품 재고 엔티티)에 접근하여 값을 읽고 수정하는 **레이스 컨디션(Race Condition)**이 원인.
    * 스레드 A가 재고가 100개인 것을 보고 99개로 줄이려는 찰나, 스레드 B 역시 동일한 100개 상태의 데이터를 읽어가서 99개로 덮어쓰게 되면서 먼저 처리된 스레드의 수정 사항이 유실.

* **해결 방법**:
    * **낙관적 락(Optimistic Lock)**과 데이터베이스 레벨에서 트랜잭션을 정렬하는 **비관적 락(Pessimistic Lock)**의 장단점을 분석함.
    * 데이터 정합성이 절대적이고 충돌이 빈번한 도메인 특성에 맞춰, 선점형으로 데이터베이스 단에서 정직하게 트랜잭션을 줄 세우는 **비관적 락 전략(`PESSIMISTIC_WRITE`)을 최종 채택**.
