# 🛒 Commerce Service
> **Spring Boot와 Spring Data JPA를 활용한 이커머스 핵심 백엔드 기능 구현 프로젝트**

## 🚀 Key Objectives
* **계층형 아키텍처**: Controller-Service-Repository 구조 기반의 책임 분리 및 비즈니스 로직 독립성 확보.
* **데이터 영속화**: 인터페이스 기반 설계를 바탕으로 인메모리 저장소에서 **MySQL로의 안정적인 전환 및 영속화**.
* **트랜잭션 정합성**: `@Transactional`을 활용한 서비스 레이어의 원자성 확보 및 데이터 제어.
* **방어적 설계**: 백엔드 코어 단의 유효성 검증을 통한 재고 부족 및 데이터 왜곡 차단.
* **동시성 제어**: 멀티스레드 환경을 시뮬레이션하여 데이터 정합성 문제를 애플리케이션 및 DB 레벨에서 해결.

## 🛠 Tech Stack
* **Framework**: Spring Boot 3.x
* **Language**: Java 17
* **ORM & Database**: Spring Data JPA, MySQL 8.x
* **Build**: Gradle
* **Template Engine**: Thymeleaf

## 📌 Main Features & Technical Achievements

### 1. 세션 기반 인증 및 권한(Role) 인프라 구축
* **인증 구조 개선**: 쿠키 위조 위험을 차단하기 위해 서버 메모리에 무작위 UUID 세션 Key를 매핑하는 구조로 고도화.
* **공통 인가 분리**: `HandlerInterceptor`를 도입하여 비로그인 사용자의 접근을 라우팅 단계에서 차단하고 컨트롤러의 중복 책임 제거.
* **역할 기반 권한 관리**: 일반 사용자의 공지사항 접근은 '조회'로 제한하고, 등록·수정·삭제는 컨트롤러 가드 레이어에서 **오직 최고 관리자(`MemberType.ADMIN`)만 가능하도록 제어**.

### 2. 주문/장바구니 도메인 정합성 및 최적화
* **트랜잭션 원자성**: '주문 생성'과 '장바구니 비우기'를 단일 트랜잭션으로 묶어 예외 발생 시 전 과정 롤백 보장.
* **영속성 최적화**: 무분별한 `Update` 쿼리를 지양하고, JPA의 **변경 감지(Dirty Checking)**를 활용해 도메인의 상태 변화를 영속성 컨텍스트를 통해 안전하게 반영.

### 3. 멀티스레드 동시성(Concurrency) 검증
* **Race Condition 시뮬레이션**: `ExecutorService`와 `CountDownLatch`를 활용하여 **100명의 사용자가 동시에 주문하는 분산 환경 테스트 케이스 구축**.
* **재고 무결성 확보**: 동시 다발적 요청 시 발생하는 데이터 유실 및 재고 음수(-) 현상을 방어하기 위해 동시성 제어 전략 적용.

## 🚨 Troubleshooting

### 1. 다중 요청(Race Condition) 발생 시 재고 데이터 무결성 붕괴
* **문제 상황**
    * 100개의 주문 요청이 동시에 인입될 때, 최종 남은 재고가 0이 아니라 80~90개로 남는 **데이터 유실(Lost Update)** 현상 발생.
* **원인 분석**
    * 여러 스레드가 하나의 공유 자원(상품 재고 엔티티)에 동시 접근하여 수정 전 값을 읽고 덮어쓰는 **레이스 컨디션(Race Condition)**이 원인.
    * 스레드 A의 수정 사항이 완료되기 전 스레드 B가 과거 재고 상태를 읽어가며 수정 사항이 유실됨을 확인.
* **해결 방법 및 성과**
    * 데이터 경합이 빈번하고 정합성이 절대적인 이커머스 도메인 특성을 고려하여, DB 레벨에서 트랜잭션을 정직하게 줄 세우는 **비관적 락 전략(`PESSIMISTIC_WRITE`)을 채택하여 정합성 100% 확보**.
    * **[추가 고도화]** 비관적 락 적용으로 발생할 수 있는 대기 타임아웃 및 커넥션 풀 고갈 문제를 방어하기 위해, **JPA의 쿼리 힌트(`jakarta.persistence.lock.timeout`)를 설정하여 락 획득 대기 시간을 제한하는 방어적 코드 추가**.

### 2. 연관 엔티티 대량 조회(findAll) 시 N+1 쿼리 폭탄으로 성능 저하
* **문제 상황**
    * 서비스 운영 중 주문 관리자 페이지 및 고객 주문 내역 화면에서 주문 목록을 전체 조회할 때, 데이터가 늘어남에 따라 API 응답 속도가 저하되는 현상 발생.
    * 데이터베이스 콘솔 확인 결과, 주문 1건을 조회하는 단일 쿼리 뒤에 각 주문과 연관된 회원(Member) 및 주문상품(OrderItem) 데이터를 조회하는 `SELECT` 쿼리가 데이터 개수(N)만큼 연쇄적으로 발생하는 **N+1 문제**를 포착함.

* **원인 분석**
    * **무분별한 즉시 로딩(EAGER)**: 엔티티 설계 시 연관 관계 전략을 고려하지 않으면 발생할 수 있는 문제로, 당장 필요 없는 연관 데이터까지 SQL에서 한번에 조회하느라 커넥션 풀이 낭비.
    * **지연 로딩(LAZY) 환경에서의 루프(Loop) 호출**: 연관 관계를 `LAZY`로 설정하더라도, 조회된 부모 엔티티 리스트를 순회하며 자식 엔티티의 필드(예: `order.getMember().getUsername()`)를 건드리는 순간 영속성 컨텍스트가 가짜 객체(Proxy)를 초기화하기 위해 추가 쿼리를 N번 더 날리게 됨을 확인.

* **해결 방법 및 성과**
    * **글로벌 지연 로딩 전략(LAZY) 통일**: 시스템 내 모든 연관 관계를 기본적으로 `LAZY`로 강제, 원치 않는 시점에 데이터베이스에 부하가 가는 것을 차단.
    * **ToOne 관계 최적화 (Fetch Join)**: 주문과 회원(`@ManyToOne`)처럼 연관된 데이터가 단 건인 경우, Repository 레이어에 커스텀 JPQL을 작성하여 **`join fetch` 문법을 적용**. 단 1방의 `JOIN` SQL로 데이터를 영속성 컨텍스트에 한 번에 얹어 쿼리 수를 **N+1번에서 단 1번으로 압축**.
    * **ToMany 컬렉션 페이징 최적화 (Batch Size)**: 1:N 관계인 주문상품(`@OneToMany` List) 구조에서 페치 조인을 쓰면 데이터 뻥튀기로 인해 DB 페이징(`limit`, `offset`) 처리가 불가능해지는 메모리 과부하 문제를 겪음. 이를 위해 **`default_batch_fetch_size: 100` 설정을 도입**. 부모 엔티티만 안전하게 페이징 조회한 뒤, 자식 컬렉션에 접근할 때 **`IN` 절 1방으로 최대 100개씩 묶어서 대량 청구**하도록 고도화.
