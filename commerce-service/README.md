# 🛒 Commerce Service
> **Spring Boot와 Spring Data JPA 기반 이커머스 핵심 백엔드 시스템**

## 🚀 Key Objectives
* **계층형 아키텍처**: Controller-Service-Repository 간 책임 분리로 비즈니스 로직의 독립성 및 유지보수성 극대화.
* **데이터 정합성**: `@Transactional`을 활용한 원자성 보장 및 멀티스레드 환경의 동시성 이슈 원천 차단.
* **성능 최적화**: JPA 인프라의 메커니즘을 이해, 조회 성능 저하 요인(N+1)을 분석 및 해결.
* **중앙 집중형 보안 및 예외 처리**: 구조적인 인증/인가 처리와 통합 예외 설계를 통한 애플리케이션 안정성 확보.

## 🛠 Tech Stack
* **Framework & Language**: Spring Boot 3.x, Java 17
* **ORM & Database**: Spring Data JPA, MySQL 8.x
* **Build & Template**: Gradle, Thymeleaf

## 📌 Main Features & Technical Achievements

### 1. 인터셉터 구조 기반 가드 패턴 및 회원 등급별 인가 정책
* **중앙 집중식 인가 분리**: `HandlerInterceptor`를 도입하여 비로그인 사용자를 원천 차단, 컨트롤러의 중복 관심사 분리.
* **세션 기반 인증**: 서버 메모리에 무작위 UUID 세션 키를 매핑하는 구조로 설계하여 쿠키 위조 방지.
* **등급별 접근 제어**: 최고 관리자(`ADMIN`)와 일반 사용자의 권한을 컨트롤러 가드 레이어에서 완벽히 분리하여 보안성 향상.

### 2. 도메인 정합성 중심의 서비스 레이어 설계
* **프로세스 원자성 확보**: '가주문 생성', '재고 차감', '장바구니 비우기' 등 복합 비즈니스 로직을 단일 트랜잭션으로 제어하여 예외 발생 시 100% 롤백 보장.
* **글로벌 예외 처리기 구축**: `@RestControllerAdvice` 기반의 예외 아키텍처를 설계하여 재고 부족, 권한 부족 등 예외 상황을 핸들링하고 클라이언트 응답 규격 통일.
* **Dirty Checking 최적화**: 무분별한 `Update` 쿼리 호출을 지양하고, JPA 영속성 컨텍스트의 변경 감지 기능을 활용해 영속 상태 데이터 변화를 안전하게 DB에 반영.

### 3. 분산 환경 동시성(Concurrency) 검증 및 제어
* **부하 테스트 시뮬레이션**: `ExecutorService`와 `CountDownLatch`를 활용해 100명의 사용자가 동시에 주문하는 레이스 컨디션 테스트 케이스 자원 구축.
* **재고 무결성 확보**: 동시성 경합 시 발생하는 재고 음수(-) 현상 및 데이터 유실을 방어하기 위해 DB 레벨의 **비관적 락 전략(`PESSIMISTIC_WRITE`)** 적용.

## 🚨 Troubleshooting

### 1. 다중 요청 발생 시 레이스 컨디션으로 인한 재고 붕괴
* **문제 상황**: 100개 동시 주문 요청 시 데이터 유실이 발생하여 최종 재고가 정합성에 맞지 않음.
* **원인 분석**: 여러 스레드가 하나의 공유 자원(상품 재고)을 동시에 읽고 덮어쓰며 수정 사항이 상쇄됨을 포착.
* **해결 및 성과**
    * 이커머스 도메인 특성을 고려해 DB 레벨에서 트랜잭션을 순차 제어하는 비관적 락 전략 도입으로 정합성 확보.
    * 락 장기 소유로 인한 커넥션 풀 고갈을 방지하고자 JPA 쿼리 힌트(`lock.timeout`)를 적용, 대기 시간을 엄격히 제한하는 방어적 코드 구축.

### 2. 연관 엔티티 대량 조회 시 N+1 쿼리 폭탄 및 페이징 과부하
* **문제 상황**: 주문 목록 전체 조회 및 타임리프 화면 렌더링 시 연관 엔티티 쿼리가 연쇄 폭발하여 응답 속도 급하강.
* **원인 분석**: 연관 관계 자식 객체 순회 및 뷰 레이어 참조 시 지연 로딩 상태의 가짜 객체(Proxy)들이 개수만큼 초기화 쿼리를 발생시킴을 확인.
* **해결 및 성과**
    * **글로벌 LAZY 통일**: 기본값이 즉시 로딩(`EAGER`)인 `@ManyToOne` 관계를 모두 `LAZY`로 강제하여 불필요한 리소스 낭비 차단.
    * **Fetch Join 최적화**: ToOne 연관 관계는 커스텀 JPQL의 `join fetch` 문법을 적용하여 단 1방의 SQL로 데이터를 묶어 가져오도록 쿼리 수 압축.
    * **Batch Size 페이징 해결**: ToMany 컬렉션 구조에서 Fetch Join 시 발생하는 페이징 과부화 문제를 예방하기 위해 `default_batch_fetch_size: 100` 도입. 부모 엔티티만 페이징 조회 후 자식 데이터는 `IN` 절로 최대 100개씩 청구하여 쿼리 효율 극대화.
